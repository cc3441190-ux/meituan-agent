# 本地场景短时活动规划与执行 Agent · 设计文档（交付版）

> 形态：React 19 + Vite WebUI；规划/工具/执行经 `core/ports` 抽象，默认 Mock 实现，可替换为美团开放平台或 LLM 后端。  
> 目标链路：自然语言输入 → 可执行行程 → 用户确认 → 并发交付 → 分享协同。

---

## 一、Planning 策略

采用**四段式分层规划**，将约束理解与供给查询解耦，保证输出可解释、可局部重规划。

| 阶段 | 入口 | 输出 | 关键参数 |
|------|------|------|----------|
| 意图解析 | `parseIntent` | `Constraints` | `timeWindow` 3–8h；`people`；`budget`；起终点 |
| 骨架生成 | `planSkeleton` | 节点序列 + 初始时间轴 | 家庭/朋友分流；≥5h 追加第 4 站 |
| 工具填充 | `MockPlanningService.createPlan` | 带 POI/库存/路线的 `Plan` | 餐饮查位；玩乐查票；相邻站 `getRoute` |
| 收敛校验 | `finalizePlan` | 冲突标注 + 时间窗合规 | `enforceTimeWindow`；`detectConflicts` |

时间轴由 `schedulePlanTimeline` 统一计算：上一站结束 → 路程 → 等位 → 缓冲 → 活动时段（含 ±15min 弹性）。超出 `timeWindow` 时优先压缩可调整站停留（下限 30min），仍超出则写入 `timeWindowOverflowMinutes` 并标注 `time_exceeded`。

用户确认阶段支持 `confirmNode`、`replanLocal`、`planTimeAdjust`；存在未处理库存/门票异常或待确认节点时，`canExecuteBookings` 禁止进入交付。

```typescript
interface Constraints {
  timeWindow: number          // 小时，3–8
  people: string[]            // family | friends | child_5 | diet | ...
  location: 'current' | 'nearby'
  preferences: string[]
  budget: 'low' | 'medium' | 'high'
  originName?: string
  destinationName?: string
}

interface Plan {
  nodes: PlanNode[]
  startTime: Date
  timeWindowOverflowMinutes?: number
}
```

## 二、工具调用链路

工具层经 `IPOIService` / `IBookingService` / `IPlanningService` 暴露；Mock 实现集中于 `mockApi.ts`，Live 模式通过 Http/Llm 适配器替换，失败时降级 Mock。

| 工具 | 签名 | 作用 | 典型返回 |
|------|------|------|----------|
| `searchPOI` | `(type, constraints) → POI` | 按评分/距离/人群/预算选点 | `id, name, rating, location` |
| `checkInventory` | `(poiId, timeSlot?) → Inventory` | 餐饮订位/排队 | `available, queue, reason` |
| `checkTicketAvailability` | `(poiId, timeSlot?) → Inventory` | 乐园/展览门票 | 同上 |
| `getRoute` | `(from, to, mode?) → Route` | 通勤估算 | `duration, mode, distance` |
| `bookDeliverable` | `(Deliverable) → BookingResult` | 订位/蛋糕/鲜花/用车等 | `orderId, eta, fallback?` |

**端到端链路**

```
用户输入
  → parseIntent → planSkeleton
  → 逐节点: searchPOI → [dining] checkInventory / [ticket] checkTicketAvailability → getRoute
  → finalizePlan(enforceTimeWindow + detectConflicts)
  ── 用户逐站 confirmNode / replanLocal ──
  → inferAddOns → Preflight 授权
  → TaskRunner.dispatch(): 拓扑分层，同层 Promise.all 并发
  → buildInviteCard → 分享 / 同伴反馈协商
```

`TaskRunner` 依据 `Deliverable.dependsOn` 构建 DAG，保证「订位成功后再送蛋糕/鲜花/备注」；同层无依赖项并行下发以缩短端到端时延（Mock 实测规划 ~1.5s，含交付 ~6–10s）。

```typescript
interface IPOIService {
  searchPOI(type: string, constraints: Constraints): Promise<POI>
  checkInventory(poiId: string, timeSlot?: Date): Promise<Inventory>
  checkTicketAvailability(poiId: string, timeSlot?: Date): Promise<Inventory>
  getRoute(from: [number, number], to: [number, number], mode?: string): Route
}

// TaskRunner 交付状态机（节选）
type DeliverableStatus =
  | 'idle' | 'queued' | 'dispatching' | 'in_progress'
  | 'done' | 'failed' | 'fallback_proposed'
```

## 三、异常处理机制

异常分为**规划期**与**执行期**两类。统一原则：检测 → 显式告警 → 提供备选/操作建议 → 等待用户确认；禁止静默替换 POI 或自动跳过失败项。

| 类型 | 代码 | 触发条件 | 用户操作 |
|------|------|----------|----------|
| 无座 | `no_seat` | 餐饮 `checkInventory.available === false` | 换同级店 / 等位改时 / 稍后处理 |
| 无票 | `no_ticket` | 玩乐 `checkTicketAvailability` 失败 | 换景点 / 改时段 / 稍后处理 |
| 行程冲突 | `time_short` | 通勤时长 > 站间空档 + 5min | 顺延 `suggestedDelay` / 代叫车 |
| 距离预警 | `too_far` | POI 距离 > 15km | 换点 / 局部重规划 |
| 时间窗溢出 | `time_exceeded` | 压缩后仍超 `timeWindow` | 删站 / 缩时 / 调整起程 |
| 交付降级 | `fallback_proposed` | `bookDeliverable` 失败且有替代 | `acceptFallback` / `rejectFallback` |
| 交付失败 | `failed` | 无可用降级 | 单项 `retry`；5min 内 `cancel` |

规划期异常经 `ExceptionAlertBar` / `InventoryNegotiationSheet` 呈现；详情页打开触发 `refreshNodeAvailability` 复查供给。执行期由 `TaskRunner` 驱动状态迁移，会话级 `sessionRef` 隔离重规划与旧任务回写；演示剧本（鲜花售罄）通过 `resetDeliverableDemoFlags` 保证可复现。

```typescript
// 规划期节点冲突字段（节选）
interface PlanNode {
  status?: 'active' | 'loading' | 'confirmed' | 'error' | 'warning'
  conflict?: 'no_seat' | 'no_ticket' | 'time_short' | 'too_far' | 'time_exceeded'
  suggestedDelay?: number
  suggestedAlternative?: string
  inventoryResolved?: boolean
}

// 执行期状态迁移（节选）
// idle → queued → dispatching → in_progress → done
//                              ↘ failed → retry
//                              ↘ fallback_proposed → acceptFallback → idle → ...
const DELIVERABLE_CANCEL_WINDOW_MS = 5 * 60 * 1000
```
