# 本地场景短时活动规划与执行 Agent · 设计文档

> 一句自然语言目标 →（规划）→ 可执行完整方案 →（确认）→ 自动完成关键下单/预订 → 一键分享给同伴。
> Demo 形态：WebUI（React 19 + Vite），全程 Mock API，可离线运行。

---

## 一、Planning 策略

采用 **「槽位解析 → 骨架生成 → 工具填充 → 冲突修正」四段式分层规划**，把「理解人」和「查供给」解耦，保证可控、可解释、可局部重规划。

1. **意图解析（parseIntent）**：从一句话里抽取结构化约束 `Constraints`：
   - 人群：`family / child_5 / diet（减肥）/ friends / group_4 / mixed_gender / couple`
   - 时间窗：默认下午 4h，识别「半天/晚上/N小时/一整天」
   - 偏好/规避：`dining / entertainment / walking / cultural / outdoor`、不吃辣、预算高低、远近
   - 起终点：正则匹配「从X出发 / 回家 / 回公司」等，生成 `originName / destinationName`
2. **骨架生成（planSkeleton）**：按主人群分流生成「起点 → 玩 → 吃 → [加站] → 终点」的节点序列，并从 14:00 起铺时间轴：
   - 家庭：乐园/室内游乐 → 轻食(减肥)/亲子餐厅 →（≥5h 加植物园散步）
   - 朋友：展览/电玩/Citywalk → 烤肉/火锅 →（≥5h 加夜市）
   - 时长自适应：`timeWindow ≥ 5` 才追加第 4 站，命中题面「下午 4–6 小时」。
3. **工具填充**：骨架里每个非固定节点依次调用 `searchPOI`（按评分+人群+预算+距离打分选店）、餐饮节点再调 `checkInventory`，相邻节点调 `getRoute` 估算通勤。
4. **冲突修正（detectConflicts）**：填充后整体扫描时间/距离/库存冲突，标注状态供前端高亮，支持用户局部「换一个/替代/调时间/删除」后 `replanLocal` 增量重算，不推翻整条路线。

> 设计取舍：规划主体是**规则驱动的确定性 Planner**（非黑盒），便于在 Demo 中稳定复现；`MockPlanningService` 已预留 `HttpPlanningService` 接口，可平滑替换为 LLM + 工具链。

## 二、工具调用链路

工具通过 `core/ports` 接口抽象，Mock 适配器实现，底层统一走 `src/agent/mockApi.ts`，便于整体替换为真实美团开放平台 API。

**工具清单（MockAPI）**

| 工具 | 作用 | 关键返回 |
|---|---|---|
| `searchPOI(type, constraints)` | 按类型+约束打分选店 | 最优 POI（评分/距离/标签） |
| `checkInventory(poiId, time)` | 查餐厅有无位置/排队 | `available`、`queue` |
| `getRoute(from, to, mode)` | 估算通勤时间/方式 | 距离、时长、walk/drive/subway |
| `bookPOI / bookDeliverable` | 下单订位/蛋糕/鲜花/用车/停车 | `orderId`、`eta`、`fallback` |

**端到端链路**

```
用户一句话
  → parseIntent(Constraints)
  → planSkeleton(骨架节点 + 时间轴)
  → 逐节点: searchPOI → [dining] checkInventory → getRoute
  → detectConflicts(标注冲突)
  ──────── 用户在卡片上确认/微调（confirmNode / replanLocal）────────
  → inferAddOns(方案+约束+原话 → 交付清单: 订位/蛋糕/鲜花/备注/用车/停车/伴手礼)
  → 用户授权
  → TaskRunner.dispatch(): 按 dependsOn 拓扑分层, 同层并发, 逐项 bookDeliverable
  → 全部完成 → 自动生成分享卡片(buildInviteCard) 发给同伴
```

要点：交付项之间存在依赖（蛋糕/鲜花/备注依赖对应「订位」先成功），`TaskRunner` 用**拓扑分层 + 同层并发**保证「先订位再送物」的正确顺序，同时尽量并行以缩短等待。

## 三、异常处理机制

异常分**规划期冲突**与**执行期失败**两类，均「提示 + 给备选 + 留人工决策」，绝不静默替换或失败即终止。

**1. 规划期冲突（detectConflicts，节点状态高亮）**
- `no_seat` 餐厅满座：标红，预备同级备选（`suggestedAlternative`），不自动替换，弹窗让用户「用备选 / 等位改时 / 忽略」。
- `time_short` 通勤时间 > 间隔：标黄并给出 `suggestedDelay` 建议顺延。
- `too_far` 距离 > 15km：标黄提示超出舒适范围。
- 库存有随机性，详情页打开会**实时复查**（`refreshSelectedInventory`），模拟真实供给波动。

**2. 执行期失败（TaskRunner，状态机驱动）**
状态流：`idle → queued → dispatching → in_progress → done / failed / fallback_proposed`
- **降级（fallback_proposed）**：如「鲜花当日售罄」，自动给出同价位替代花束，用户可一键接受（`acceptFallback`）或拒绝（`rejectFallback`）。
- **重试（retry）**：失败项可单独重试，不影响其他已成功订单。
- **可撤销窗口**：下单成功后 5 分钟内（`DELIVERABLE_CANCEL_WINDOW_MS`）可 `cancel`，给用户后悔余地。
- **会话隔离**：`sessionRef` 防止重规划时旧任务回写脏数据；演示用的「鲜花失败」剧本仅触发一次（`resetDeliverableDemoFlags`），保证 Demo 可复现。

> 失败兜底原则：任何环节出错都**收敛到一次明确的人工确认**，符合题面「帮你把事做完」而非「丢一堆搜索结果」。
